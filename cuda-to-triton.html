Complete mental model mapping for CUDA practitioners learning Triton

Recently, I started learning Triton and found that the biggest hurdle wasn't the API itself — it was rewiring my CUDA mental model. Triton doesn't think in threads; it thinks in blocks. Shared memory? Automatic. Syncthreads? Gone. Once that shift clicks, the rest follows fast.

If you're coming from CUDA and picking up Triton for kernel development — whether for LLM inference, fused ops, or custom attention — this is for you.

The Paradigm Shift

🧩

Threads → Programs

In CUDA, you think in threads. In Triton, you think in program instances. Each program processes an entire block/tile of data, not a single element.

🧠

Manual → Automatic

Shared memory, coalescing, synchronization, and tensor core scheduling are all handled by the compiler. You never write __shared__ or __syncthreads().

⚡

Scalar → Block Ops

Every operation is on a block of values (like NumPy). tl.load fetches a tile, arithmetic operates on the whole tile, tl.store writes a tile.

Side by Side: Vector Addition

CUDA
__global__ void add_kernel(
    float* x, float* y,
    float* out, int N) {

  // Each thread: one element
  int idx = blockIdx.x * blockDim.x
            + threadIdx.x;

  // Bounds check
  if (idx < N) {
    out[idx] = x[idx] + y[idx];
  }
}

// Launch
add_kernel<<<(N+255)/256, 256>>>(
    x, y, out, N);
Triton
@triton.jit
def add_kernel(
    x_ptr, y_ptr, out_ptr, N,
    BLOCK: tl.constexpr):

  # Each program: BLOCK elements
  pid = tl.program_id(0)
  offs = pid * BLOCK + tl.arange(0, BLOCK)

  # Mask replaces if-statement
  mask = offs < N
  x = tl.load(x_ptr + offs, mask=mask)
  y = tl.load(y_ptr + offs, mask=mask)
  tl.store(out_ptr + offs, x + y, mask=mask)

# Launch
grid = (triton.cdiv(N, 1024),)
add_kernel[grid](x, y, out, N, BLOCK=1024)

Side by Side: Row Softmax (Reduction Pattern)

CUDA — 50+ lines typical
// Need: shared memory for reduction,
// warp shuffles for final reduce,
// two passes (max then exp/sum),
// __syncthreads() between passes,
// manual coalescing of row loads,
// bank conflict avoidance...

__shared__ float smax[256];
__shared__ float ssum[256];

// Pass 1: find max (tree reduction)
// Pass 2: exp(x - max) and sum
// Pass 3: normalize by sum
// ... ~60 lines of careful code
Triton — Clean & complete
@triton.jit
def softmax_kernel(
    in_ptr, out_ptr, n_cols,
    stride, BLOCK: tl.constexpr):

  row = tl.program_id(0)
  offs = tl.arange(0, BLOCK)
  mask = offs < n_cols

  # Load entire row as a block
  ptrs = in_ptr + row * stride + offs
  x = tl.load(ptrs, mask=mask,
              other=-float('inf'))

  # All reductions: one-liners!
  x_max = tl.max(x, axis=0)
  x = tl.exp(x - x_max)
  x_sum = tl.sum(x, axis=0)
  out = x / x_sum

  out_ptrs = out_ptr + row * stride + offs
  tl.store(out_ptrs, out, mask=mask)

Triton API Quick Reference

tl.program_id(axis)

Returns the ID of the current program instance along the given axis (0, 1, or 2). Like blockIdx.x/y/z in CUDA.

tl.arange(start, end)

Returns a 1D block of consecutive values [start, end). Used to compute offsets for the current program's tile.

tl.load(ptr, mask, other)

Loads a block of data from memory. mask handles bounds; other is the default for masked-out elements.

tl.store(ptr, val, mask)

Stores a block of data to memory. Only elements where mask=True are written.

tl.dot(a, b)

Block-level matrix multiply. Auto-uses tensor cores when shapes are multiples of 16. Returns a 2D block.

tl.sum / tl.max / tl.min

Block-level reductions along an axis. Replaces CUDA's warp shuffles + tree reduction. Single-line reductions!

tl.where(cond, x, y)

Element-wise ternary: returns x where cond is True, y otherwise. Replaces CUDA's divergent branches.

tl.constexpr

Marks a parameter as compile-time constant. The kernel is recompiled for each unique value — enables aggressive optimization.

tl.exp / tl.log / tl.sqrt / tl.abs

Element-wise math ops on blocks. Work exactly like their CUDA counterparts but on entire tiles at once.

tl.atomic_add(ptr, val)

Atomic addition to global memory. Same semantics as CUDA's atomicAdd.

@triton.autotune

Decorator that tests multiple configurations (BLOCK_SIZE, num_warps, num_stages) and picks the fastest one per input shape.

triton.testing.do_bench(fn)

Reliable GPU benchmarking with proper warmup and timing. Returns median runtime in milliseconds.

⚠ Key Constraints to Remember:
• Block sizes must be powers of 2 (64, 128, 256, 512, 1024...)
• tl.dot needs shapes as multiples of 16 for tensor cores
• No thread-level control — you cannot do warp-level primitives (__shfl_sync)
• No explicit shared memory — the compiler decides what goes in SRAM
• All pointers are raw — you compute addresses with pointer arithmetic
• Kernel is recompiled for each unique set of tl.constexpr values

The Triton Launch Pattern

Standard Launch Pattern
import torch
import triton
import triton.language as tl

# 1. Define kernel
@triton.jit
def my_kernel(in_ptr, out_ptr, N, BLOCK: tl.constexpr):
    pid = tl.program_id(0)
    offs = pid * BLOCK + tl.arange(0, BLOCK)
    mask = offs < N
    x = tl.load(in_ptr + offs, mask=mask)
    # ... compute ...
    tl.store(out_ptr + offs, x, mask=mask)

# 2. Wrapper function (handles tensor → pointer)
def my_op(x: torch.Tensor) -> torch.Tensor:
    out = torch.empty_like(x)
    N = x.numel()
    BLOCK = 1024
    grid = (triton.cdiv(N, BLOCK),)  # ceil(N / BLOCK)
    my_kernel[grid](x, out, N, BLOCK=BLOCK)
    return out

# 3. Use it
x = torch.randn(10000, device='cuda')
y = my_op(x)
